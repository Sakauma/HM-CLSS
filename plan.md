# HM-CLSS 航行情绪层升级计划

## 目标

在不改变现有主功能结构的前提下，为 HM-CLSS 增加一层“克制但高级”的航行情绪反馈，让系统从“功能完整”升级为“更有人机温度、更有记忆点、但不打扰使用”。

本轮改造只做小而精的升级，不扩展导航，不新增主页面，不重做已有模块逻辑。


## 现阶段设计原则

- 保留当前《挽救计划》主题，不引入跳脱世界观的新视觉语言。
- 不新增独立介绍卡，不恢复会挤占首屏可操作面积的大型 banner。
- 视觉变化以微光、色温、文案、局部质感和轻量动画为主，不靠重型特效堆砌。
- 所有新增反馈必须服务状态表达，不能只是装饰。
- 保持纯前端、零构建、`localStorage` 为核心存储的现状不变。
- 优先兼容现有暗黑模式和移动端布局。
- 避免紫色系偏移，继续沿用当前工程已经建立的蓝金、深空、舰桥气质。
- 高风险操作必须按意图拆分入口，默认入口优先处理“今天正在发生”的事情。
- 涉及历史修正或未来预填报的流程，必须有单独 UI、单独校验和明确标签。
- 所有补录类行为都必须留痕，并且在统计、状态和连续打卡口径上保持可解释。


## 实施范围

### 1. 全站环境态引擎

新增一个全局派生状态 `voyageAmbientState`，只读，不改写现有业务数据。

环境态固定为 4 种：

- `steady`：正常巡航
- `alert`：异常预警
- `recovery`：恢复推进
- `nightwatch`：夜航值守

状态来源固定为以下现有数据：

- 当前本地时间
- 当日打卡状态
- 当前任务完成度
- 深度工作是否开启
- 调酒模块当前情绪倾向

默认映射规则：

- 白天且状态稳定，进入 `steady`
- 出现打卡警告、任务积压或关键状态异常，进入 `alert`
- 已开始修复推进、深度工作进行中、或调酒倾向偏恢复时，进入 `recovery`
- 夜间时段统一进入 `nightwatch`

实现要求：

- 用统一派生函数集中计算，不允许各页面各自拼条件。
- 缺失某类数据时按保守回退处理，不报错、不阻断页面渲染。
- 状态变更只影响表现层，不改变统计逻辑和原始记录。


### 2. 全站环境表现层

基于 `voyageAmbientState`，统一驱动全站轻量视觉反馈。

具体改造：

- 顶部主 banner 增加状态相关的副标题、微光强度、纹理层级和背景色温偏移。
- 左侧导航当前页高亮增加更明确的“在线/工作中”反馈，但不做高频闪烁。
- 页面背景允许做轻微的亮度、透明度、粒子噪点或渐变偏移，但不改变现有布局骨架。
- 所有反馈都必须遵守低打扰原则，首屏重点仍然是功能卡片而不是视觉特效。

表现约束：

- 新增动画只允许使用轻量位移、透明度和渐变变化。
- 必须兼容暗黑模式。
- 必须支持 `prefers-reduced-motion`，必要时退化为静态表现。


### 3. 页面级微叙事文案

给关键页面增加 1 到 2 行动态状态文案，全部嵌入现有区域，不新建介绍卡。

页面范围：

- 首页/总览：在主状态区下方补一行系统播报式文案。
- 任务管理：在深度工作相关区域加入状态短句，补足当前空荡感。
- 舰桥日志：为空状态和归档状态补一条符合主题的系统提示。
- 离舰审批：把说明性文案改得更直接、更人机友好。
- 调酒吧台：完成态结果文案增加更短更准的服务语，不新增操作负担。

文案要求：

- 每条文案必须可由当前状态推导，不能写成固定装饰语。
- 不允许长段解释，不允许出现“欢迎使用”“探索更多”一类空话。
- 文案语气统一为舰桥系统播报风格，克制、明确、略带叙事感。


### 4. 低打扰彩蛋系统

加入一组默认隐藏的小惊喜，只在满足条件时轻量出现。

彩蛋触发场景固定为三类：

- 连续多日打卡合格后，首页或调酒页出现一次“航程纪念句”。
- 夜间进入调酒吧台时，触发夜航版高光或一句额外服务语。
- 完成关键任务组合时，在日志页出现一次性恢复进度提示。

触发规则：

- 单次会话最多触发一次。
- 同类彩蛋 24 小时内不重复。
- 不使用弹窗，不强制打断操作，不遮挡主按钮。
- 彩蛋只允许落在现有卡片、角标、辅助文案或轻视觉状态中。


### 5. 偏好项与回退机制

新增一个轻量本地偏好存储项：

- `hmclss_ambient_preferences`

默认结构固定为：

```json
{
  "enabled": true,
  "intensity": "subtle",
  "easterEggs": true
}
```

约束：

- 偏好缺失时自动回退到默认值。
- 关闭 `enabled` 后，全站应退回当前基础视觉，不影响功能使用。
- 不做复杂设置中心，只预留状态层的开关能力。


### 6. 离舰流程拆分与风险控制

将当前“一个日期输入框同时处理正常请假、补请假、预请假”的混合表单拆成 3 条独立流程，避免误触。

固定拆分为：

- `今日离舰`：只处理今天，默认入口，隐藏日期选择器。
- `预请假`：只处理未来日期，单独表单。
- `补请假`：只处理过去日期，单独表单，强调修正属性。

交互要求：

- 三条流程使用分段切换或标签式入口，不允许继续共用一套模糊文案。
- `今日离舰` 默认聚焦，日期固定为今天，不允许改成过去或未来。
- `预请假` 只能选择未来日期，并在提交前明确展示目标日期。
- `补请假` 只能选择过去日期，必须额外填写“补录说明”。
- 若目标日期已经存在打卡或离舰记录，`补请假` 需要二次确认，避免误覆盖。
- 历史记录列表需要用清晰标签区分 `正常`、`预请假`、`补请假`。

数据要求：

- `leaveData` 记录补充 `requestMode`，固定值为 `normal | planned | retro`。
- 同步补充 `createdAt` 和 `correctionNote` 字段，便于后续解释记录来源。
- 继续复用现有 `checkinData[date]` 联动方式，但要保证过去和未来记录不会误刷新今日状态。


### 7. 补打卡机制、配额与留痕

新增一条单独的“补打卡”流程，不能复用当前实时打卡按钮。

范围与限制固定为：

- 仅允许补录过去 30 天内的记录。
- 不允许补今天，也不允许补未来。
- 以“日期”为配额单位，同一天内补多个班次只计 1 次补录。
- 最近 7 天最多允许补录 2 天。
- 每个自然月最多允许补录 6 天。

交互要求：

- 补打卡入口单独放在值班模块的辅助操作区，不放进三张实时班次卡片里。
- 表单必须包含：目标日期、班次、连线时间、登出时间、补录原因。
- 提交前展示“状态预判”，明确这次补录会落成 `合规 / 警告 / 豁免 / 异常` 中的哪一种。
- 如果目标班次已有记录，必须先展示现有内容，再由用户确认是否覆盖。
- 如果目标日期已是全天离舰，则禁止补打卡并提示先修正离舰记录。
- 如果目标日期存在分段离舰，则继续沿用当前豁免判定逻辑。

留痕要求：

- 补录后的班次记录必须带 `entrySource` 标记，固定值为 `live | retro`。
- 需要补充 `updatedAt` 和 `correctionReason`，确保后续统计和排查可解释。
- 在表格或历史视图中，对补录生成轻量 `补录` 标签，不隐藏修正来源。


### 8. 统计、状态与成就口径联动

补请假、预请假和补打卡都必须联动刷新统计和摘要，避免出现“记录变了但面板没变”的割裂。

联动要求：

- 首页总览、今日状态、离舰记录、打卡表格都要识别新增的流程来源。
- 补打卡成功后，需要同步刷新合规率、班次统计和顶部摘要数字。
- 补请假和补打卡都要重新计算连续打卡，避免 streak 仍沿用旧结果。
- 未来日期的 `预请假` 不应污染今天的状态判断，但在到达目标日时必须能正确接管。
- 历史修正不会触发当天实时打卡按钮状态变化，除非目标日期就是今天。


## 实施顺序

### Phase 1. 状态层基础

- 补齐环境态派生函数
- 定义统一状态枚举和默认回退
- 将状态源与现有打卡、任务、调酒数据接通

### Phase 2. 全站表现层联动

- 给主 banner、背景层、导航高亮接入环境态
- 保证暗黑模式和移动端不回退
- 接入 `prefers-reduced-motion`

### Phase 3. 页面微叙事补强

- 首页、任务、日志、离舰、调酒五个区域逐个补充状态文案
- 修正深度工作区域的视觉留白问题
- 保持首屏卡片可直接使用，不引入新的下滚负担

### Phase 4. 离舰流程拆分

- 将离舰模块重构为 `今日离舰 / 预请假 / 补请假` 三段式交互
- 补齐日期边界校验、风险确认和记录标签
- 保证默认入口只服务当天离舰，不再混用历史和未来日期

### Phase 5. 补打卡治理

- 新增补打卡入口、表单和状态预判
- 接入 30 天范围限制、7 天 / 月度配额限制和覆盖确认
- 给补录结果增加来源标记和修正说明

### Phase 6. 彩蛋收尾

- 加入一次性、低频、可回退的彩蛋逻辑
- 给彩蛋增加本地冷却记录
- 保证不影响主流程和主要统计

### Phase 7. 质量校验

- 做脚本语法检查
- 做桌面端、移动端、暗黑模式和低动态偏好验证
- 做关键状态切换冒烟测试
- 做补请假 / 预请假 / 补打卡的限额与覆盖校验
- 做统计、首页摘要和 streak 联动校验


## 验收标准

- 首屏重点仍然是功能卡片，新增情绪层不会抢主交互。
- 环境态在不同时间和不同任务/打卡状态下能稳定变化。
- 暗黑模式正常，新增样式不会导致页面渲染中断或颜色失控。
- 新增文案不会挤压按钮、输入区或造成卡片高度失衡。
- 彩蛋不会重复刷屏，不会弹窗打断，不会造成信息噪音。
- 关闭情绪层偏好后，页面回退到当前基础状态且功能不受影响。
- `今日离舰` 入口不能误提交过去或未来日期。
- `预请假` 和 `补请假` 在列表中有明确标签，且不会与正常请假混淆。
- 补打卡超出 7 天或月度配额时会被明确拦截，不会静默失败。
- 补打卡覆盖已有记录时，必须经过确认，且结果会带补录标记。
- 补请假和补打卡完成后，统计、首页摘要和连续打卡结果同步更新。


## 不在本轮范围内

- 新增主模块或新页面
- 引入音效
- 重做数据结构或同步协议
- 重型 WebGL、Canvas 大动画
- 引入多人审批、后端审核或云端权限系统
- 改写现有调酒和任务的核心业务规则
