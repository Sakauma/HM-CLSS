# HM-CLSS UI / UX 升级规划

## 目标
在现有功能与模块拆分已基本稳定的前提下，后续升级以“更人机友好、更高级、但不破坏《挽救计划》主题”为唯一主线。

- 保持当前零构建架构和原生前端技术栈。
- 先解决信息层级、易用性和视觉一致性，再做装饰性润色。
- 以小步快跑的方式推进，每次改动都可独立验证和回退。

## 体验北极星
后续所有 UI / UX 方案都应同时满足下面四条：

- `3 秒可读`
  用户进入任一主面板后，应能在 3 秒内看懂当前状态、主任务和下一步操作。
- `1 步可达`
  今日最常用动作，例如打卡、开始任务、记录速记、查看同步状态，应尽量在 1 次点击或 1 次滚动内到达。
- `反馈明确`
  用户执行保存、完成、失败、删除、同步后，必须立即收到明确反馈，不依赖猜测。
- `主题不失真`
  视觉升级后的系统仍然要像“深空维生终端”，而不是通用 SaaS 后台，也不是纯炫技赛博面板。

## 设计方向
### 视觉基调
- 采用“冷静舰桥 + 有温度的舱内光”方向。
- 主背景以深空蓝、石墨灰、雾面金属色为主。
- 强调色以信号青、暖琥珀为主，告警红仅用于异常和危险操作。
- 避免大面积霓虹紫、廉价玻璃拟态、过量描边发光和过密技术装饰线。

### 排版与信息密度
- 中文正文优先保证可读性，标题可以略带技术感，但不应牺牲扫描效率。
- 数据、时间、编号、状态值可以局部使用等宽风格，正文不要大面积等宽化。
- 高密度信息面板应通过层级、留白和标签来减压，而不是简单缩小字号。

### 交互与动效
- 动效只服务于状态变化、层级切换和反馈确认。
- 常规动效时长控制在 150ms 到 300ms，避免浮夸拖沓。
- 任何关键流程都不能依赖 hover 才能成立，移动端必须有等价体验。

### 文案与世界观
- 保留“维生系统”“解除休眠”“离舰报备”等设定词。
- 首次接触即可理解的功能说明必须同步存在，避免纯设定化表达。
- 世界观用于增强沉浸感，不用于增加理解负担。

## 非目标
- 不进行框架迁移，不引入新的构建流程。
- 不为了“更科幻”而做纯 HUD 化重皮肤。
- 不在同一次迭代里同时做大规模重构、功能新增和全站视觉翻新。

## 阶段规划
### Phase 0：现状审计与基线冻结
目标：先确认哪里丑、哪里难用、哪里不一致，避免一上来凭感觉改。

- 盘点所有主面板在桌面端、移动端、浅色、深色下的当前截图。
- 标记信息层级混乱、空状态缺失、按钮样式不统一、文本难懂、交互反馈不足的问题。
- 整理出高频路径：今日打卡、开始任务、记录速记、请假、查看统计、同步数据。

交付物：

- 一份界面问题清单。
- 一组现状截图基线，供后续 before / after 对比。

验收标准：

- 每个主模块至少有一张桌面截图和一张移动截图。
- 高频路径的起点、终点和阻塞点被明确记录。

### Phase 1：视觉令牌与基础组件统一
目标：先打地基，解决“同一产品像多个页面拼起来”的问题。

- 统一颜色角色：背景层、卡片层、边框层、主文本、次文本、强调色、成功、警告、危险。
- 统一圆角、边框、阴影、分组标题、区块间距。
- 统一按钮、输入框、下拉框、标签、空状态、Toast、Modal、表格 / 卡片列表的基础样式。
- 明确浅色与深色的对应关系，避免仅靠“反色”完成暗色模式。
- 选定中文优先的字体策略，并限制等宽字体的使用范围。

交付物：

- 一套可复用的视觉令牌和基础组件规则。
- 至少 2 到 3 个模块完成基础样式换皮验证。

验收标准：

- 不同模块中的按钮、卡片、输入、状态标签具有一致规则。
- 深色和浅色下的关键文本、图表和状态色都可读。
- 后续模块改造时不需要重新发明按钮和卡片样式。

### Phase 2：信息架构与全局导航优化
目标：让系统先“好找、好懂”，再谈高级感。

- 重整首屏或总览区，让今日状态、关键提醒、主操作前置。
- 优化导航层级、当前态高亮、模块命名和分组逻辑。
- 把次要说明、历史记录、进阶统计做渐进展开，减少首屏噪音。
- 为高频动作设计稳定的“快速入口”，减少来回跳转。

交付物：

- 更清晰的首页 / 总览结构。
- 更稳定的导航与当前态表达。

验收标准：

- 用户无需滚动太久就能找到今日关键动作。
- “我在哪个模块”“现在该做什么”在桌面端和移动端都足够清晰。

### Phase 3：核心高频模块逐个升级
目标：优先处理真正每天会用到的界面。

优先顺序：

1. 航星维生日志与今日状态
2. 核心科研阵列
3. 噬星体捕捉池
4. 休眠与离舰报备
5. 深空通讯链路
6. 全舰效能雷达与深空酒馆

模块升级重点：

- 航星维生日志与今日状态
  强化班次状态、时间窗口、当前结果和下一步操作的可见性。
- 核心科研阵列
  突出进行中的任务、计时状态、任务输入入口和历史记录层级。
- 噬星体捕捉池
  降低记录阻力，增强分类、时间信息和回看体验。
- 休眠与离舰报备
  让请假范围、影响时段和当前生效状态一眼可懂。
- 深空通讯链路
  明确本地 / 云端状态、最近同步结果、失败原因和危险操作边界。
- 全舰效能雷达与深空酒馆
  提升图表可读性、图例清晰度、情绪记录的氛围与信息组织。

验收标准：

- 每次只改一个模块或一条高频流程，避免过大 PR。
- 模块升级后，首要动作更快找到，历史信息更易扫描，状态反馈更明确。

### Phase 4：微交互、反馈与可访问性补强
目标：把“能用”提升到“顺手”和“可信”。

- 补齐保存成功、删除成功、同步完成、失败重试等反馈链路。
- 为危险操作加确认机制，必要时提供可撤销或二次确认。
- 补齐键盘焦点、ARIA 语义、标题层级和仅颜色区分的风险点。
- 为窄屏布局检查点击热区、滚动路径和内容溢出。
- 为动效增加 `prefers-reduced-motion` 兜底。

交付物：

- 一套统一的反馈模式和交互细节规则。
- 一轮移动端与可访问性修正。

验收标准：

- 不存在关键操作“点了但不知道有没有成功”的情况。
- 375px 视口下无横向溢出，高频按钮可稳定点击。
- 键盘用户可以看见焦点位置，危险动作不会被误触直接执行。

### Phase 5：整体抛光与发布前验收
目标：收尾一致性问题，形成稳定设计基线。

- 回查所有模块的颜色、字重、间距、标题层级、按钮优先级是否一致。
- 修正零散异常态、空状态、长文本、超长列表和图表边界情况。
- 更新 README 中如有必要的界面截图和说明。
- 固化新的设计基线，避免后续迭代再次回到“模块各改各的”。

交付物：

- 一套可持续维护的 UI / UX 基线。
- 一轮完整手动验收记录。

验收标准：

- 各模块视觉语言一致，不再出现明显“拼装感”。
- 关键路径在桌面端、移动端、浅色、深色下都能稳定完成。

## 建议执行节奏
- 每次迭代尽量聚焦一个 Phase 下的一个子目标。
- 单次 PR 最好只处理一个模块或一类组件，不要把视觉重构和功能调整混在一起。
- 每次提交都保留 before / after 截图，以及对应的手动验证步骤。

## 最终验收清单
满足以下条件时，可认为本轮 UI / UX 升级达到目标：

- 用户进入系统后，能快速理解“今天状态”和“下一步动作”。
- 打卡、开始任务、记录速记、请假、同步等高频操作都足够顺手。
- 所有空状态、成功态、失败态和危险操作都有明确反馈。
- 浅色和深色模式都具备稳定对比度与一致氛围。
- 移动端不只是“能缩下去”，而是真的保住了核心可用性。
- 整体视觉更克制、更成熟，但仍然一眼能认出这是《挽救计划》主题的维生系统。
