# 仓库指南

## 项目结构与模块组织
这个仓库是一个无需构建步骤的静态前端应用，但代码已经从单文件逐步拆分为“页面壳 + 模块脚本”的结构。

- `index.html`
  页面骨架、Tailwind CDN 配置、自定义 CSS、各模块挂载点，以及少量必须与 DOM 紧贴的结构性标记。
- `assets/js/core.js`
  共享常量、数据模型、`localStorage` 读写、通用工具函数。
- `assets/js/theme.js`
  明暗主题切换与全局视觉状态同步。
- `assets/js/navigation.js`
  面板切换、导航高亮与入口路由。
- `assets/js/checkin.js`、`leave.js`、`tasks.js`、`notes.js`、`tavern.js`、`stats.js`、`sync.js`、`status-ui.js`、`phone-achievements.js`
  按业务域拆分的初始化、渲染与交互逻辑。

修改时请遵守以下边界：

- UI 标记仍应放在 `index.html` 对应区域附近，不要把结构分散到难以追踪的位置。
- 共享常量、标签映射、成就配置和跨模块工具优先归拢到 `assets/js/core.js`。
- 新增交互优先扩展现有 `init*`、`render*`、`update*` 模式，避免把多个模块重新耦合回单一脚本。
- 如果某个 UI 升级只影响单一业务域，请优先在对应模块内完成，不要顺手做无关重构。

## 构建、测试与开发命令
目前项目没有包管理器或正式构建流水线，常用命令如下。

- `python3 -m http.server 8000`
  在 `http://localhost:8000` 启动本地静态服务器，用于浏览器验证。
- `open index.html` 或直接拖入浏览器
  用于快速视觉冒烟测试。
- `node --check assets/js/*.js`
  对拆分后的脚本做基础语法巡检。
- `rg "keyword" index.html assets/js`
  编辑前优先定位页面区块、样式钩子与事件逻辑。

## 编码风格与命名规范
保持当前技术栈不变：HTML、Tailwind 工具类、少量自定义 CSS 和原生 JavaScript。

- 使用 4 个空格缩进、分号，以及描述性的 camelCase 函数名，例如 `updateTodayStatus`、`initTaskManagement`。
- 只有真正稳定的常量才使用 `UPPER_SNAKE_CASE`。
- 新增状态类、提示文案和按钮标签时，优先复用现有语义，而不是同义词横飞。
- 面向用户的文案继续保持“中文科幻维生系统”语境，但可读性优先，避免为了设定感牺牲理解成本。
- 视觉样式调整应优先收敛为可复用的颜色、间距、阴影、圆角与状态规则，不要在多个模块里复制零散 Tailwind 组合。

## UI / UX 设计守则
后续 UI 和 UX 升级都必须以“更人机友好、更高级，但不破坏《挽救计划》主题”为前提。

- 主题定位
  保持“深空维生终端 / 舰桥工作台”气质，而不是滑向夸张游戏 HUD 或廉价赛博朋克皮肤。
- 视觉基调
  优先使用深空蓝、石墨灰、舱内暖琥珀、信号青等克制配色，保留告警红的稀缺性。避免大面积霓虹紫、廉价玻璃拟态和过度发光描边。
- 高级感来源
  依赖清晰的信息层级、克制的光影、稳定的间距节奏、统一的卡片与表单规范，而不是堆特效。
- 人机友好
  所有关键区域都要先回答“我现在是什么状态”“下一步最重要的操作是什么”“系统刚刚发生了什么”。
- 文案策略
  可以保留“解除休眠”“离舰报备”这类世界观词汇，但旁边要给出足够直白的说明，让首次使用者不需要猜。
- 交互策略
  关键动作尽量一屏可达；次要信息用渐进展开处理；删除、覆盖、同步冲突等高风险操作必须给出清晰确认或结果反馈。
- 反馈规范
  保存、完成、失败、同步、删除后都应有即时视觉反馈；不要让用户靠猜测判断是否成功。
- 空状态规范
  所有空状态都要有解释和下一步动作，不能留下“空白大卡片”。
- 响应式策略
  窄屏下优先保留今日状态、主按钮、输入区和最近记录；高密度表格或统计面板要能合理降级成卡片或纵向堆叠。
- 可访问性
  正文与关键状态需保持足够对比度，键盘焦点可见，动效遵守 `prefers-reduced-motion`，不要仅靠颜色表达结果。
- 动效原则
  只保留少量有意义的转场与状态动画，时长以 150ms 到 300ms 为主，强调“系统响应”而非“装饰炫技”。

## 测试指南
当前仓库没有自动化测试。所有 UI / UX 相关修改至少需要手动验证以下内容：

- 面板之间的导航切换与当前态高亮。
- `localStorage` 持久化、刷新后的状态恢复，以及 Gist 同步入口是否仍清晰可用。
- 受改动影响的任务、打卡、请假、速记、酒馆、图表等完整流程。
- 明暗主题切换后的对比度、图表可读性和状态色表达。
- 空状态、成功提示、失败提示、删除确认等反馈是否完整。
- 窄屏设备上的布局、按钮可点区域、内容溢出和滚动体验。
- 至少覆盖 375px、768px、1024px 三档视口；重大视觉改动建议补看 1440px。

如果某次修改主要针对 UI 或 UX，请在 PR 描述中补充：

- 改动目标与设计意图。
- 用户可见变化。
- 手动验证步骤。
- 对应截图或录屏，至少包含桌面端和移动端各一张。

## 提交与 Pull Request 规范
最近提交历史使用简洁的 Conventional Commit 前缀，例如 `feat:`、`fix:`、`refactor:`。请保持这一格式，例如：`feat: refine mission control card hierarchy`。

Pull Request 应聚焦单一主题，不要把视觉润色、功能新增和无关重构混在一起。涉及设计升级时，请额外说明：

- 本次改动改善了哪个用户问题。
- 是否引入新的视觉令牌、交互模式或组件规则。
- 是否影响现有手动测试路径。

## 安全与配置提示
应用支持通过 GitHub Gist 同步。不要把个人令牌、Gist ID 或导出的私密数据提交到仓库中，也不要出现在截图、录屏或示例配置里。
